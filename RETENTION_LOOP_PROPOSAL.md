# Witness Retention Loop Proposal

## Current Core Loop

```text
Iris Hub
→ Continue Witness Memory
→ Witness Moment
→ completion
→ local Resonance record
→ reflective Iris return
→ Iris Hub
```

The current foundation makes completion persistent without changing the protected moment flow.

## Proposed Retention Systems

These are product proposals only. They are not implemented by the current profile foundation.

### Daily Witness

A single authored Memory Moment selected for a calendar day.

Requirements before implementation:

- authored daily eligibility data
- deterministic date policy
- completion/replay policy
- offline behavior
- optional local streak policy

### Archive

A local presentation of completed moment records.

Requirements before implementation:

- archive navigation surface in Iris Hub
- use `WitnessProfile.completed_moment_ids` only
- no new content or duplicate registry

### Mastery

A replay layer that records real future result data and awards optional mastery Resonance.

Requirements before implementation:

- runtime result payload with actual accuracy/anomaly data
- explicit assistance policy
- visible replay result presentation

### Discovery

A future authored-content expansion surface.

Requirements before implementation:

- new authored moments
- registry expansion policy
- Memory Field presentation rules

## Retention Principles

- Return because a memory remains unfinished, not because a timer forces a task.
- Reward attention and mastery with real result data.
- Keep daily and archive systems inside the Iris Hub fantasy.
- Do not add store, ads, battle pass, social pressure, or artificial scarcity.
- Do not generate content automatically; future moments remain authored.
